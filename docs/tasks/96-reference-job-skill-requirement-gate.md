## B6. 代表シナリオによる確認

- [x] 入出力と再実行方針が明確な DB chunk ジョブで、実装開始を選択できることを確認する
- [x] 軽微な名称やサンプル値だけが未定の依頼で、暫定試行を選択できることを確認する
- [x] ファイル再実行時の重複防止方式が未定の依頼で、設計確認を選択できることを確認する
- [x] 外部 API、partition、非同期処理を含む依頼を既存パターンへ無理に当てはめないことを確認する
- [x] 質問が網羅的な要件票にならず、判断に必要な少数へ絞られていることを確認する

確認結果:

| シナリオ | 判定 | 確認内容 |
| --- | --- | --- |
| 入出力、commit、失敗時 rollback、先頭からの安全な rerun が明確な DB 更新 | `実装開始` | `customer-import` を基準に DB chunk の実装へ進める |
| ジョブ名や架空サンプル値だけが未定の小規模な参照ジョブ | `暫定試行` | 暫定名称とサンプル値を示し、局所的に変更可能な範囲へ限定する |
| ファイルから DB へ更新するが、再実行時の重複防止が未定 | `設計確認` | 重複防止、失敗後の再開方式、ファイル移動時点に絞って確認する |
| 外部 API、partition、非同期処理を含み、復旧・重複排除方式が未定 | `設計確認` | 既存3パターンへ分類せず、実行モデルと復旧方式の設計検証を先行する |

検証中に、Skill の事前確認と作業手順が既存3パターンからの選択を必須と読める箇所を検出した。既存パターンの組み合わせまたは非適合も判断できる表現へ修正した。

設計確認の質問は、実装判断を変える事項を原則1から3件に絞る。上記 `File → DB` の例でも、網羅的な要件票ではなく、整合性と復旧を決める3事項を優先する。
